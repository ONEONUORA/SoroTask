# Debugging with Execution Traces

This document explains how to use correlation IDs to trace a task's lifecycle through the SoroTask Keeper backend.

## Overview

Every task processing flow is assigned a unique `correlationId`. This ID allows you to connect logs from the initial discovery (polling) through simulation, submission, and final outcome handling.

### Trace Stages

1.  **Polling Cycle**: Groups multiple task checks under a `cycle-XXXX` ID.
2.  **Task Discovery**: Each task check gets a `poll-TASKID-XXXX` ID.
3.  **Task Execution**: When a task is enqueued for execution, it is assigned an `attemptId` (e.g., `task-TASKID-TIMESTAMP-XXXX`). This ID serves as the primary `correlationId` for the simulation and submission stages.

## How to Search Logs

### 1. Identify a Problem Task
If a task fails, find its `taskId` in the logs.

```bash
grep "taskId\":123" keeper.log
```

### 2. Follow the Execution Trace
Look for the `correlationId` or `attemptId` associated with that task in the `executeTask failed` or `Transaction finalised` log lines.

Example log line:
```json
{"level":"info","module":"executor","correlationId":"task-123-1714350000000-abcd","message":"Transaction submitted","taskId":123,"txHash":"..."}
```

### 3. Trace Back to Polling
You can find which polling cycle discovered the task by searching for the `poll-123-XXXX` ID which is linked in the `task:skipped` or `task:started` events in the queue logs.

```bash
grep "task-123-1714350000000-abcd" keeper.log
```

## Correlation ID Format

- **Cycles**: `cycle-[hash]` (e.g., `cycle-a1b2c3d4`)
- **Polling**: `poll-[taskId]-[hash]` (e.g., `poll-123-e5f6g7h8`)
- **Execution**: `task-[taskId]-[timestamp]-[hash]` (e.g., `task-123-1714350000000-i9j0k1l2`)

## Key Log Fields

- `module`: The component generating the log (e.g., `poller`, `executor`, `queue`).
- `correlationId`: The unique trace ID for the current operation.
- `taskId`: The ID of the task being processed.
- `attemptId`: The specific attempt ID generated by the idempotency guard (used as correlation ID during execution).

## Debugging Tips

- **Simulation Failures**: Search for the `correlationId` to see the exact state of the task when simulation was attempted.
- **Duplicate Executions**: Traces will show if multiple polling cycles discovered the same task and how the idempotency guard handled them.
- **Latency Analysis**: Use the timestamps of logs with the same `correlationId` to calculate the time spent in each stage (Poll -> Queue -> Simulate -> Submit -> Finalise).
